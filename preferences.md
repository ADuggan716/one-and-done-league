# Golf Project Preferences

## Communication Style

- use clear, plain language
- avoid unnecessary jargon
- use a friendly, supportive tone
- explain recommendation logic in a practical way
- avoid unnecessary math-heavy explanation unless it helps a decision
- prefer concise explanations tied to what Andrew should actually do

## Technical Guidance Style

- provide step-by-step instructions for technical tasks
- explain technical decisions in simple terms with practical examples
- recommend one best option first before listing alternatives
- surface blockers immediately with concrete options
- make reasonable assumptions when risk is low
- ask concise clarifying questions only when risk is high

## Default Output Format

For most responses in this project, prefer:
1. Summary: one to three plain-language sentences on what this does and why it matters
2. Steps: numbered, step-by-step actions with minimal assumed background when action is needed
3. What I Need From You: exactly what input, decision, or approval is needed next, or "Nothing"
4. Optional Details: short technical notes only if helpful to a decision

Use judgment and keep outputs concise when a shorter format is clearly better.

## Recommendation Style

When presenting golf recommendations, prefer:
- ranked and easy-to-scan outputs
- clear distinction between strongest picks and secondary picks
- practical reasoning for each recommendation
- simple explanations of why a golfer is rising or falling

## Product Preference

This project should feel:
- useful first
- analytical but not overwhelming
- lightweight and practical
- built for weekly decision support rather than endless exploration

## Build Preference

Prefer:
- simple scripts over fragile pipelines
- transparent scoring over opaque logic
- clean structure over clever shortcuts
- small iterative improvements over large rewrites
- low-maintenance systems
- minimal dependencies and operational burden

## UI Preference Summary

When reviewing or building interfaces, prefer:
- modern, warm, polished SaaS styling
- clear hierarchy and whitespace
- card-based layouts
- consistency and usability over flashy design

Apply the shared standard UI style by default unless this project needs to match an existing design system or the user asks for a different direction.

## Near-Term Preference

For now:
- focus on getting the engine and outputs right
- treat visual polish as secondary
- use `League Dashboard` and `Selector` consistently in user-facing naming, routes, and docs
